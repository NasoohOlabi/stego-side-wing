StegoPipeline — src/workflows/pipelines/stego.py

Report generated for code review: what this module does, how it fits the workflow, and why wall-clock time is dominated by remote work (LLM + HTTP), not by the bit-stego codec.

1. Role in the system

StegoPipeline is the sender-side orchestrator: it embeds a secret payload into Reddit-style post data so the output looks like ordinary comments and angle metadata, while remaining decodable by DecodePipeline in src/workflows/pipelines/decode.py. It intentionally mirrors the legacy n8n workflow workflows/27rZrYtywu3k9e7Q.json (workflow id STEGO_WORKFLOW_ID).

Stateful dependencies (injected in __init__):

  • BackendAPIAdapter — posts I/O, needle_finder_batch, artifact save
  • LLMAdapter — OpenAI-compatible chat to LM Studio (provider="lm_studio")
  • DecodePipeline — used only for cross-validation during encode (not for the final user-facing decode path)

2. Main entry points

encode(payload, post, tag, max_retries=4)

Core algorithm (comments in source at stego.py ~427–441):

  1. Post augmentation — pure codec: compress payload, choose comment chain and angle indices, produce commentEmbedding / angleEmbedding (workflows.utils.stego_codec.augment_post).
  2. Sample construction — up to four candidate angles get best_match text via needle_finder_batch over post["search_results"].
  3. Candidate generation — for each sample, call the LLM to emit exactly three JSON strings (stego comment texts).
  4. Validation — treat the first sample’s three strings as primary candidates; remaining samples supply few-shot context. For each primary string, run DecodePipeline.decode; success if decoded angle matches the selected angle.
  5. Retry loop — on failure, repeat up to max_retries + 1 attempts (default 5 total attempts).

process_post(...)

CLI/API-style wrapper: resolves payload (or reads default from workflow JSON SetSecretData), optionally picks the next unprocessed post id, loads post JSON from disk, calls encode, then on success writes an n8n-shaped artifact via save_object_local.

3. Data flow (high level)

post + payload
    → augment_post (codec, local CPU)
    → needle_finder_batch (HTTP or local semantic match × up to 4 needles)
    → FOR each sample (1..4): LLM encode (sequential)
    → FOR each of 3 primary strings: decode() = semantic_search + LLM (× up to 5 tries)
    → retry entire attempt if validation fails

4. Why it is slow

Latency is not from the steganographic bit packing (dictionary build, compression, comment/angle embedding). Those run in-process and are typically small compared to the following.

Primary bottleneck: many sequential LLM round-trips per successful encode, each subject to model speed, context length, and LLMAdapter retries/backoff (LLM_MAX_ATTEMPTS, exponential backoff + jitter).

4.1 Encode-phase LLM calls scale with sample count

In encode, the loop at stego.py ~493–505 calls _generate_stego_texts for every entry in samples. Samples are built from up to four angles (totalAnglesSelectedFirst[:4] in _build_samples). Each iteration:

  • LLM call_llm with max_tokens=STEGO_ENCODE_MAX_TOKENS (1536) — large headroom for models that emit “thinking” before JSON.
  • temperature=0.7 — more variance and often longer generations than greedy decoding.

So one encode attempt issues 1 × (number of samples) full chat completions, with samples ≤ 4 → up to 4 encode LLM calls per attempt.

4.2 Validation multiplies cost: 3 × full decode pipelines per attempt

_cross_validate loops over the three primary candidate strings and calls self.decode_pipeline.decode(...) for each. Each decode (decode.py ~144–377):

  • NET backend.semantic_search with n=DECODE_SEMANTIC_TOP_N (20) over the full tangents_db / angles list.
  • LLM Up to DECODE_LLM_MAX_TRIES (5) completion calls on failure before giving up.

Per encode attempt, that is roughly 3 × (1 semantic search + 1–5 LLM calls) in the worst case.

4.3 Outer retry loop

max_retries=4 means the whole block (all sample encodes + all cross-validations) can run up to 5 times when validation fails or exceptions occur. Failed attempts still paid for LLM and search work.

4.4 Needle finder batch

_build_samples calls needle_finder_batch once per attempt with up to four needles. Via HTTP, the client uses a 60s timeout (backend_api.py). Locally, the adapter runs find_best_match sequentially per needle — still potentially expensive if each match embeds or scores against a large haystack.

4.5 Logging volume

The pipeline logs full system and user prompts and raw LLM output at INFO. That does not usually dominate latency but increases I/O and log size during long runs.

5. Rough operation budget (single encode attempt)

Let S = number of samples (1–4), R = decode LLM tries (1–5 per candidate). Order-of-magnitude:

StepCalls (typical)Kind
augment_post1CPU in-process
needle_finder_batch1NET or CPU × needles
_generate_stego_textsSLLM sequential
decode (per primary string)3each: semantic + LLM (× R)

Example with S=4 and three decode paths each needing 2 LLM tries: 4 encode + 3 semantic + 6 decode LLM calls — before any outer retry. Multiply by up to 5 for the retry loop.

Takeaway: To make encode faster, reduce sequential LLM work (fewer samples, smaller max_tokens, parallel encode where safe), reduce decode work during validation (cheaper model, fewer retries, or skip decode for dev), or shrink the retry surface — the codec path is already the cheap part.
