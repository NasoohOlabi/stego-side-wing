Research pipeline report

Source: src/workflows/pipelines/research.py · Generated for performance analysis.

1. Purpose

ResearchPipeline enriches a post with web research: it generates search queries from the post, runs web search for each query, deduplicates and filters hits (skips PDFs and duplicate links), fetches page text for the selected URLs, and stores the concatenated fetched texts in post["search_results"]. It mirrors legacy n8n semantics for deciding whether a post is “new” (needs research) via _is_new_post.

2. High-level flow

preview_post / research_post
    │
    ├─► Skip if post already has non-empty search_results (unless force)
    │
    ├─► GenSearchTermsPipeline.preview_generation  →  LLM or SQLite cache
    │
    ├─► FOR each search term (sequential):
    │       Google CSE (or Bing fallback on quota-style errors)
    │       → up to 10 results/term, dedupe links, drop .pdf
    │
    └─► FOR each batch of 3 unique URLs (parallel within batch):
            ThreadPoolExecutor(3) × per-URL _fetch_url_with_timeout_retries
                → FetchUrlContentPipeline.fetch → ContentAdapter → crawl4ai stack

5. Why it is slow (root causes)

The pipeline is I/O-bound and structured so wall-clock time grows with the number of terms, URLs, and worst-case fetch behavior.

5.1 Sequential web search per term

  • High impact. The loop over search_terms runs one _web_search_google_or_bing call after another. If you have T terms, search phase time is roughly the sum of T HTTP latencies (plus any key-rotation retries on Google), not the max.
  • Each call requests up to 10 results (count=10), increasing downstream URL count.

5.2 Bounded URL concurrency (batch size 3)

  • High impact. Unique selected links are fetched in batches of 3 (batch_size = 3). Total fetch-wall time scales about as ceil(U/3) × (slowest fetch in each batch) where U is unique URLs after deduplication.
  • Many distinct domains × slow pages makes this the dominant cost for research-heavy posts.

5.3 Per-URL timeout and retries

  • Medium–high. Default RESEARCH_FETCH_TIMEOUT_SEC is 180 seconds per attempt; RESEARCH_FETCH_RETRIES defaults to 1 (two attempts total). A stuck crawl can cost up to ~360s for that URL before failure.
  • Each attempt wraps fetch in a fresh ThreadPoolExecutor(max_workers=1) to enforce future.result(timeout=…). That adds thread churn and does not make the underlying fetch faster.

5.4 Expensive fetch stack (crawl4ai + LLM)

  • High impact. fetch_url_content_crawl4ai uses crawl4ai + LLM extraction on cache miss. Headless or crawler paths are inherently slower than a plain HTTP GET of static HTML.
  • Without a warm disk cache (use_fetch_cache=False or cold cache), every URL pays full external latency.

5.5 Validation-driven refetch storms

  • Medium. FetchUrlContentPipeline.fetch may perform up to 3 additional uncached fetches if the first body fails validate_content. A batch of 3 URLs could multiply into many sequential live fetches inside worker threads.

5.6 LLM cold start for terms

  • Medium (first run per post). Cache miss triggers LLMAdapter.call_llm (LM Studio path in metadata). Model load, queueing, and token generation add seconds to minutes depending on hardware and model size.

5.7 Multi-post processing is serial

  • Medium at scale. process_post_objects processes posts one after another. Throughput for N posts is N × (single-post research time), not parallelized at the pipeline level.

5.8 Google key fallback chain

  • Variable. search_google tries GOOGLE_API_KEY_1 … _5 sequentially on errors. Worst case adds multiple 20s timeouts (per key attempt) before success or Bing fallback.

6. Order-of-magnitude model (rough)

Illustrative only — use logs (research_google_query_done, research_fetch_batch_done) for real numbers.

T_terms   = number of search terms
U_urls    = unique non-PDF links after dedupe
t_search  ≈ mean Google/Bing latency per term
t_fetch   ≈ mean successful fetch time (dominated by crawl4ai)

Time_search  ≈ T_terms × t_search          (sequential)
Time_fetch   ≈ ceil(U_urls / 3) × t_fetch  (3-wide parallelism, batch barrier)

Worst URL    ≈ (1 + RESEARCH_FETCH_RETRIES) × RESEARCH_FETCH_TIMEOUT_SEC
               (+ validation refetches inside fetch pipeline)
