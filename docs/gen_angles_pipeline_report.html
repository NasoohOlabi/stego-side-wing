GenAnglesPipeline — what it does and why it is slow

Report for src/workflows/pipelines/gen_angles.py and its primary dependency chain (angles service → angle runner → LM Studio HTTP).

1. Role in the system

GenAnglesPipeline enriches a post with angles: structured items with source_quote, tangent, category, and optionally source_document (index into the text dictionary). It is used by the workflow runner, receiver pipeline, API routes, and CLI.

Stateful responsibility: holds BackendAPIAdapter, LLMAdapter, workflow config, and _last_batch_summary for batch runs.

4. Fallback path (when primary analysis fails)

If analyze_angles raises and allow_fallback=True, _generate_angles_llm joins all dictionary strings with \n\n---\n\n into one prompt and calls LLMAdapter.call_llm once. That single call can be extremely large and slow (or hit context limits), and source_document is set to 0 for all items.

5. Why the primary path is slow (detailed)

The bottleneck is not Python CPU work; it is sequential remote/local LLM inference over HTTP, scaled by how many text blocks and batches a post produces.

5.1 Outer loop: one iteration per dictionary string

In angle_runner.analyze_angles_from_texts, the code walks texts in order. For a Reddit-style post with many comments, build_post_text_dictionary produces one entry per comment body (plus selftext and search snippets). Example: 1,800+ strings means 1,800+ sequential passes through the loop body (minus empty strings).

Critical: There is no parallelism across comments or search results. Wall time grows roughly linearly with the number of dictionary entries (times LLM latency per entry).

5.2 Per string: chunking and batching → multiple LLM calls per string

  • Long text is split with _chunk_text_at_boundaries (max MAX_CHARS_PER_TEXT = 30,000).
  • Segments are grouped into batches capped by _effective_max_chars_per_prompt() (default 80,000 characters unless ANGLES_MAX_CHARS_PER_PROMPT is set).
  • Each batch triggers _run_angle_llm_on_batch → HTTP POST to LM Studio chat completions.

So one very long comment can still become multiple LLM requests; many short comments still mean one request per comment (each batch often holds one segment).

5.3 Each LLM call: retries, logging, and failure expansion

  • Retries: ANGLES_LLM_MAX_ATTEMPTS defaults to 6. Transient HTTP statuses (429, 5xx) and some connection errors trigger exponential backoff (ANGLES_LLM_RETRY_BACKOFF_BASE_SEC default 1.5s, cap 60s) via time.sleep.
  • Timeout: If ANGLES_LLM_REQUEST_TIMEOUT is unset, requests.post uses timeout=None — a stuck server can block indefinitely.
  • Context errors: On responses that look like context/token limits, the runner splits the combined batch into CONTEXT_RETRY_NUM_CHUNKS (default 3) overlapping parts and runs the LLM once per part — multiplying work.
  • Transport splitting: On certain request errors, batches are bisected recursively (depth up to ANGLES_TRANSPORT_SPLIT_MAX_DEPTH, default 24), each sub-batch invoking the LLM again.
  • JSON repair: Invalid JSON or schema issues trigger additional LLM calls (_repair_json) with a separate system prompt.
  • Every batch appends the full user prompt to prompts.log on disk.

5.4 Cache behavior (helps repeat runs, not first run)

Cache key is SHA256(text) per dictionary string. A cache hit skips LLM work for that string; a miss runs all batches. First-time processing of a large thread remains slow.

5.5 Batch workflows

process_post_objects processes posts strictly one after another (each including LLM work + save_post_local). Batch wall time is the sum of per-post times.

7. Environment knobs (angle runner)

Relevant variables from src/pipelines/angles/angle_runner.py (defaults in parentheses):

  • ANGLES_MODEL / MODEL — which model LM Studio loads (inference speed varies).
  • ANGLES_MAX_CHARS_PER_PROMPT (80_000) — larger batches = fewer calls but heavier each call.
  • ANGLES_LLM_MAX_ATTEMPTS (6), ANGLES_LLM_RETRY_BACKOFF_* — retry aggressiveness.
  • ANGLES_LLM_REQUEST_TIMEOUT — unset = no timeout.
  • ANGLES_CONTEXT_RETRY_CHUNKS (3), ANGLES_CONTEXT_RETRY_OVERLAP_CHARS (5000).
  • ANGLES_TRANSPORT_SPLIT_MAX_DEPTH (24), ANGLES_MIN_SEGMENT_SPLIT_CHARS (4000).

8. Practical takeaways

  1. Many comments ⇒ many sequential LLM round-trips — this is the dominant structural reason gen_angles feels slow on large threads.
  2. First run pays full cost; cache only helps when identical text blocks reappear.
  3. Operational issues (5xx, context limits, malformed JSON) trigger extra calls and sleeps, stretching runs further.
  4. Fallback mode avoids per-comment HTTP fan-out but sends one enormous prompt — often worse for latency and reliability.
