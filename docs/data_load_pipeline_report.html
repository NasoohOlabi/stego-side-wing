DataLoad pipeline report

Module: src/workflows/pipelines/data_load.py — how it works and why wall-clock time is dominated by URL fetching, not Python iteration.

1. Role in the system

DataLoadPipeline bridges “posts that only have a URL” to “posts whose article body is available as selftext.” It is the sender workflow stage that materializes page text before research and angle generation.

Step contract (filter-url-unresolved) — from infrastructure config:

  • Reads post JSON from datasets/news_cleaned (source directory for the step).
  • Writes enriched posts to datasets/news_url_fetched (destination directory).

Downstream code (e.g. WorkflowRunner.run_data_load, receiver rebuild paths) calls this pipeline to ensure each post has fetched article text when needed.

3. Public methods (summary)

4. Control flow: process_posts

  1. List filenames — posts_list(step, count, offset) returns fileNames (unprocessed JSON names under the step’s source dir, sorted by file size descending).
  2. Outer batch loop — slices file_names into chunks of batch_size (default 5 from runner).
  3. Per file (sequential) — get_post_local → read url → fetch_pipeline.fetch(url, use_cache=True) → if success and non-empty text, append to batch_results.
  4. Per successful post (sequential) — save_post_local(post, step="filter-url-unresolved") writes to the destination directory.

Note: batch_size does not parallelize network I/O; it only groups work for structure. Every URL fetch runs one after another inside each batch.

5. Why it is slow (root causes)

5.1 Dominant cost: each URL is a full content pipeline

Every post with a URL triggers FetchUrlContentPipeline.fetch, which delegates to ContentAdapter.fetch_url_content. On a cache miss, the implementation calls fetch_url_content_crawl4ai, which:

  • Runs crawl4ai with structured LLM extraction (extract_structured_data, configured model in analysis service). That path launches async browser/LLM work and is orders of magnitude slower than reading a local file.

Even with use_cache=True, the first visit to a URL pays this full price; subsequent runs can be much faster if both the adapter-level cache and datasets/url_cache layers contain a valid result.

5.2 Validation retries multiply live fetches

FetchUrlContentPipeline runs a lightweight heuristic validation on fetched text (ContentAdapter.validate_content: error-page indicator strings). If the cached or first result fails validation, the code calls _recover_with_fresh_fetches, which performs up to three additional fetches with use_cache=False, bypassing disk cache reads — i.e. up to four full fetch attempts per URL in the worst case.

Effect: “Bad” or noisy HTML (many error keywords) can turn one logical URL into several sequential crawl4ai round-trips, dramatically increasing latency for that post.

5.3 No concurrency

process_posts uses nested for loops with blocking calls. Wall time for N posts with cold cache is roughly sum of individual fetch latencies, not max or average. Throughput is limited to one in-flight URL fetch at a time.

5.4 Listing posts can touch many files

list_posts builds the list of unprocessed files by comparing source vs destination directories (see posts_service). For very large datasets this adds filesystem overhead and sorts by size, but this is usually minor compared to Section 5.1–5.3.

5.5 Disk writes per success

Each successful enrichment triggers a full JSON rewrite (save_post_local). This is cheap relative to crawling but adds sequential I/O after each batch’s successful fetches.

6. Scaling intuition

ScenarioTypical behavior
Warm URL cache, validation passesMostly local JSON read + cache read + write — fast per post.
Cold cache, crawl4ai + LLMVery slow per URL; dominates batch runtime.
Validation failuresUp to 4× sequential fetches per URL.
